# Meta CAPI + Attribution Setup — Website Template (v1.0)

## Purpose

This document describes the Meta Pixel + Conversions API (CAPI) + attribution
system built into Houzflow's CONFIG-driven multi-page website templates
(painters, epoxy/concrete coatings, and future niches: HVAC, plumbing,
roofing, etc.). It supersedes the original quiz-funnel SOPs
(`META-CAPI-SOP-V1.md` / `V2.md`) for **website** deployments.

**Feed this entire file to Claude when:**
- Building a new niche template (HVAC, plumbing, roofing, etc.) that should
  match this architecture
- Auditing or debugging an existing template's attribution setup
- Wiring the GHL/Make.com automation that consumes the webhook payload

---

## Relationship to the Quiz Funnel SOPs (v1/v2)

| Layer | Quiz Funnel (v1/v2) | Website Template (this doc) |
|---|---|---|
| Client-side Pixel + attribution capture | Built once per quiz, manually validated | **Built into `components.js`, ships with every template — zero per-client code edits** |
| Hidden form fields | Manually added to one `<form>` | Auto-injected by `buildLeadForm()` into every form on every page |
| Pixel base code placement | Manually pasted into `<head>` | Auto-injected into all ~25-30 HTML files via `CONFIG.metaPixelId`, gated so it's a no-op until set |
| Hashing (em/ph/fn/ln) | Server-side, via `HashingUtils` in the GHL automation script | **Unchanged** — still server-side. The website sends raw values only. |
| GHL custom fields (24 fields) | Created via `CUSTOM_FIELDS_SNAPSHOT` | **Unchanged** — same field list, same snapshot-and-reuse pattern |
| GHL workflows (Lead/Schedule/Purchase → CAPI) | `WorkflowsSnapshot` factory functions | **Unchanged** — same three workflows, same factory pattern |
| Event dedup (`lead_event_id`) | Generated client-side, stored in sessionStorage | **Same mechanism**, generalized to fire on every form on every page (not just one quiz form) |

**Bottom line:** v1/v2's Phases 1–4 (Meta Pixel/CAPI creation, GHL custom
fields, hashing, workflows) apply **unchanged** to website deployments. Only
Phase 5 (client-side code injection) is different — and for the website
template, Phase 5 is **already done** and ships in the template. Per-client
setup is now just two CONFIG.js values.

## Per-Client Setup (Website)

There is no Phase 5 code injection step for new clients. Setup is:

1. **Meta side** (manual, via Meta Ads MCP or Events Manager) — same as v1/v2 Phase 1:
   - Create Pixel for the client
   - Generate CAPI access token
   - Note the Pixel ID (16 digits)

2. **CONFIG.js** (the only file edited per client):
   ```js
   webhookUrl:  "https://hook.make.com/your-webhook-id",
   metaPixelId: "1234567890123456",
   ```

3. **GHL automation** — run v2's `setup:v2` script against this client's
   location (custom fields, hashing utils, three workflows). This is
   identical to the quiz-funnel process — the website's webhook payload
   (below) is a superset of what those workflows expect.

That's the entire per-client checklist for tracking. No HTML/JS edits.

## Architecture: How It Works

### 1. Pixel base code (`<head>`, every page)

Every HTML file loads `CONFIG.js` in `<head>` (moved here specifically so
the Pixel snippet can read `CONFIG.metaPixelId` before the page renders),
followed by an inline snippet that:

- No-ops entirely if `CONFIG.metaPixelId` is empty
- Otherwise loads `fbevents.js`, calls `fbq('init', CONFIG.metaPixelId)`,
  fires `fbq('track', 'PageView')`, and injects the `<noscript>` fallback

This means the Pixel is "always there but dormant" until a client's
`metaPixelId` is set — safe for staging/preview deployments.

### 2. Attribution capture (`components.js`, every page)

`HouzflowAttribution` (an IIFE in `components.js`) runs on every page load
and captures:

| Field | Source |
|---|---|
| `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`, `utm_term` | URL query params |
| `ad_id`, `adset_id`, `campaign_id` | URL query params (populate via ad URL templates) |
| `fbclid` | URL query param |
| `fbp` | `_fbp` cookie (set by the Pixel) |
| `fbc` | `_fbc` cookie, or derived from `fbclid` + timestamp if cookie not yet set |
| `source_url` | `window.location.href` |
| `user_agent` | `navigator.userAgent` |
| `lead_event_id` | Generated fresh per page load: `evt_{timestamp}_{random}` |

- Current session's data → `sessionStorage['houzflow_attribution']`
- First-touch snapshot (source/campaign/ad_id/timestamp) → `localStorage['houzflow_first_touch']`, written only once (never overwritten)

### 3. Hidden form fields (`buildLeadForm()`)

Every form generated by `buildLeadForm()` — home page mini-forms, contact
page full form, service page forms, city page forms — includes 14 hidden
inputs (one per attribution field above) via `attributionHiddenFieldsHTML()`.

### 4. Submit handlers (`initLeadForm()`, chat widget)

On submit:
1. Hidden fields populated from `HouzflowAttribution.get()`
2. Full payload (visible + hidden fields) POSTed to `CONFIG.webhookUrl`
   (`fetch`, `mode: 'no-cors'`, no CORS setup needed)
3. `fbq('track', 'Lead', {}, { eventID: lead_event_id })` fires — **same
   `lead_event_id`** as sent in the webhook payload, enabling Pixel↔CAPI
   deduplication on Meta's side

## Webhook Payload Reference

```json
{
  "source":        "lead-form",
  "name":          "John Smith",
  "phone":         "5551234567",
  "email":         "john@example.com",
  "service":       "<service-slug>",
  "city":          "<city-slug>",
  "sqft":          "600",
  "message":       "...",
  "smsConsent":    "yes",

  "utm_source":    "facebook",
  "utm_medium":    "paid",
  "utm_campaign":  "spring-promo",
  "utm_content":   "",
  "utm_term":      "",
  "ad_id":         "120210000000000",
  "adset_id":      "",
  "campaign_id":   "",
  "fbclid":        "IwAR...",
  "fbp":           "fb.1.1234567890.987654321",
  "fbc":           "fb.1.1234567890.IwAR...",
  "source_url":    "https://client-site.com/?utm_source=facebook&fbclid=IwAR...",
  "user_agent":    "Mozilla/5.0 (...)",
  "lead_event_id": "evt_1718000000000_a1b2c3d4e"
}
```

Notes:
- `"source"` is `"lead-form"` or `"chat"`.
- `email` only present on the contact page form (by design — see
  "New Niche Checklist" below for why it's not on every form).
- `city` only present on city-page forms.
- All `utm_*`/`fb*`/attribution fields are **always present** (empty string
  if not applicable), so the GHL workflow can map them to custom fields
  unconditionally.

### Mapping to GHL Custom Fields (v1/v2 Phase 2 field list)

The 24-field `CUSTOM_FIELDS_SNAPSHOT` from the quiz-funnel SOP maps directly:
`utm_source` → `utm_source`, `fbclid` → `fbclid`, `fbp` → `fbp`, `fbc` → `fbc`,
`source_url` → `source_url`, `user_agent` → `user_agent`,
`lead_event_id` → `lead_event_id`, `ad_id`/`adset_id`/`campaign_id` →
matching fields. No new fields are needed for the website.

### Server-side hashing reminder

The website sends **raw** `name`, `phone`, `email`. The GHL workflow's
`HashingUtils` (SHA-256, lowercase+trim for email/name, digits-only +
country code for phone) produces `em`/`ph`/`fn`/`ln` for the CAPI call.
Do not hash client-side.

## New Niche Checklist (HVAC, Plumbing, Roofing, etc.)

When building a new niche template from this architecture:

1. **Copy `components.js` verbatim** — `HouzflowAttribution`,
   `attributionHiddenFieldsHTML()`, `buildLeadForm()`, `initLeadForm()`, and
   the chat widget's submit handler are niche-agnostic. No changes needed.

2. **Add `metaPixelId: ""` to `CONFIG.js`** alongside `webhookUrl`.

3. **Inject the `<head>` Pixel snippet into every HTML file**:
   - Move the existing `<script src="CONFIG.js">` tag from bottom-of-body
     into `<head>` (must load before the Pixel snippet, since the snippet
     reads `CONFIG.metaPixelId` — note this is the unqualified top-level
     `CONFIG` binding, not `window.CONFIG`)
   - Insert the Pixel snippet immediately after, before any other `<head>`
     content
   - Relative path to `CONFIG.js` depends on file depth (`CONFIG.js` at
     root, `../CONFIG.js` one level deep, etc.) — `generate-pages.js`'s
     `fixPaths()` already handles this for generated pages as long as the
     `pages/*.html` source templates use the correct relative depth
   - Apply to **every** HTML file: root pages, `pages/*.html` source
     templates, and any already-generated `services/*.html` / `cities/*.html`

4. **Fix the service dropdown placeholder copy** in `buildLeadForm()` —
   currently reads "Type of Painting Project" (painters) / "Type of Flooring
   Project" (epoxy). For a new niche, change to match
   (e.g. "Type of HVAC Service", "Type of Roofing Project").

5. **Email field**: only the contact page form passes `showEmail: true`.
   Keep this convention — adding email to every micro-form increases
   friction without meaningfully improving Meta match quality, since the
   contact form already captures it for the highest-intent leads.

6. **Verify no double `CONFIG.js` declaration** — `CONFIG.js` uses
   `const CONFIG = {...}`. It must be loaded by exactly **one** `<script>`
   tag per page (now in `<head>`). Loading it twice throws
   `SyntaxError: Identifier 'CONFIG' has already been declared` and breaks
   the entire page.

7. **Smoke test**: load `?utm_source=facebook&fbclid=test123`, submit any
   lead form, and confirm:
   - Webhook payload includes all attribution fields with `fbclid=test123`
     and a derived `fbc`
   - `fbq('track','Lead',...)` fires in the Pixel Helper / Events Manager
     Test Events tool with a matching `event_id`

## File Manifest (per template)

- `CONFIG.js` — per-client config, including `webhookUrl` and `metaPixelId`
- `components.js` — shared logic: header, footer, chat widget, lead forms,
  `HouzflowAttribution`, Pixel-aware submit handlers
- `*.html` (all pages) — `<head>` contains CONFIG.js load + Pixel snippet
- `generate-pages.js` — regenerates `services/*.html` and `cities/*.html`
  from `CONFIG.services[]` / `CONFIG.serviceAreas[]`; preserves the injected
  `<head>` snippet via `fixPaths()` path rewriting
